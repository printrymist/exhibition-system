# Rohei Printer System への「AIネイティブ設計」応用観点

Encoreで採用された「AIネイティブ・UIはリード・オンリー」という設計思想を、Rohei Printer System(QRいいねアプリ)にどう応用できるかを整理したメモ。

---

## 前提:来場者向け機能には適用しない

Encoreの設計は **操作する人がAIを使いこなせる前提** で成り立つ。

QRいいねアプリの来場者(美術館・ギャラリーの一般客)は、スマホでQRを読んで「いいね」を押すだけのライトなユーザー。ここにAIチャット経由の操作を持ち込むのは確実に逆効果。

→ **来場者向けUIは従来通り、シンプルなボタン操作のままがベスト**

応用すべきは **ギャラリーオーナー・アーティスト・アートフェア運営者向けの管理画面** の方。

---

## 応用候補1:展示会セットアップをAIネイティブ化

**最も効果が大きそうな領域**

### 現状で想定される作業

ギャラリーオーナーが新しい展示会を始める時に発生する作業:

- 展示会情報の登録(タイトル、期間、会場)
- 作品の登録(タイトル、作家、価格、サイズ、解説)
- QRコードの生成
- キャプションの印刷設定
- 公開設定

これらを画面でポチポチ入力するのは結構な負担。

### Encoreの発想を適用すると

**ExhibitionDSL** のようなものを設計して、AIに話しかけるだけで展示会が立ち上がる:

```
[ギャラリーオーナー]
「来週月曜から2週間、田中花子さんの個展を3階ギャラリーで開催。
出品作は20点で、リストはこのExcelに入ってる。
キャプションは英日併記で、価格は会員のみ表示にして」

[AI]
→ ExhibitionDSLを生成
→ 作品一覧をExcelから読み取り
→ Firestoreに登録
→ QRコード一括生成
→ キャプションPDF作成

[ギャラリーオーナー]
管理画面で結果を確認(リード・オンリー)
```

写真や作品リストの添付ファイルをAIが解釈してくれれば、入力作業が劇的に減る。

## 応用候補2:作品情報の管理をDSL化

各作品の情報も、構造化されたDSLとして扱う発想が使える。

```yaml
artwork:
  id: tanaka-001
  title: 朝の光
  titleEn: Morning Light
  artist: 田中花子
  year: 2026
  medium: 油彩、キャンバス
  dimensions: 60x80cm
  price: 380000
  priceDisplay: members-only
  caption: |
    早朝の窓辺で...
  qrSettings:
    allowReactions: true
    allowComments: true
    reactionTypes: [love, inspired, want]
```

### メリット

- AIが自然言語の指示(「全作品の価格を10%値上げして」「英訳キャプションをまとめて生成して」)で **一括操作** できる
- GitHubでバージョン管理しやすい
- 別の展示会で作品データを再利用しやすい
- バックアップ・復元が単純なファイルコピーで済む

---

## 応用候補3:分析ダッシュボードを「AIに質問する」形に

すでに分析ダッシュボードを実装済みだが、Encore的に発想すると:

### 従来型ダッシュボード

グラフがたくさん並んでいて、ユーザーが目的のグラフを探す。

### AIネイティブな分析

ダッシュボードはあくまで「概要を見る」用(リード・オンリー的)。深掘りはAIに質問:

```
[オーナー]「先週の田中さんの展示、どの作品が一番反応良かった?」
[AI]→ 分析して回答+該当のグラフだけを表示

[オーナー]「30代女性の来場者の傾向を教えて」
[AI]→ 該当データを集計+インサイトを文章で説明
```

固定ダッシュボードを増やすより、**質問駆動型** にした方が、ギャラリーオーナーの「知りたいこと」に柔軟に応えられる。

---

## 応用候補4:アーティストプロフィールページの生成

今後の予定にある「アーティスト・プロフィールページ」も、DSLとAIの組み合わせが効きそう。

```
[アーティスト]
「私のプロフィール作って。経歴はこの履歴書PDFに書いてある。
コンセプトは自分の作品を見てAIに推測してほしい」

[AI]
→ 履歴書PDFを解析
→ 登録済み作品の画像をClaude APIで解析
→ ArtistProfileDSLを生成
→ プロフィールページを構築
```

アーティストは技術的なスキルが低い人も多いので、AI経由のセットアップは特に価値がある。

---

## 応用候補5:8月のアートフェア・デモへの効果

8月のデモmilestone を考えると、**「AIに話しかけて展示が立ち上がる」** というデモは強力。

### デモシナリオ案

審査員や見込み客の前で:

1. 何もない状態から開始
2. オーナー役が「○○の展示会を作って」と話しかける
3. 30秒後に全作品のQRコード付きキャプションが印刷可能な状態に
4. 来場者役がQRを読んでリアクション
5. オーナーがAIに「今の反応どう?」と質問→即座にインサイト

→ **「AI時代のギャラリー運営」** という強烈なストーリーになる。競合(おそらく従来型の管理画面ベースのプロダクト)との差別化が一目で伝わる。

---

## 注意点・トレードオフ

フルAIネイティブ化にはリスクもある:

1. **開発コスト**:DSLの設計とAI連携の作り込みは、普通のCRUD画面より工数がかかる
2. **AIの間違い**:登録ミスがあると展示会運営に直接影響する。確認・修正フローは必須
3. **オフライン耐性**:アートフェア会場でネットが不安定だとAI機能が動かない可能性
4. **コスト**:Claude API利用料が来場者数や操作数に比例して増える

---

## おすすめ:段階的な導入

8月のデモまでの時間を考えると、**全面リニューアルではなく、効果の高い1点突破** がよさそう。

### 優先度順

#### 最優先:「展示会セットアップAI」を1機能だけ作る

- ExhibitionDSLを設計
- 「自然言語+Excel添付で展示会が立ち上がる」体験を作る
- 既存のUIは残したまま、AI経由のセットアップを追加機能として提供
- デモで強烈なインパクトを出せる

#### 次点:作品情報のDSL化

- 既存データをDSL形式に変換できるようにする
- 一括操作(価格更新、英訳生成など)をAI経由で可能に

#### 将来:分析の質問駆動化

- 既存ダッシュボードと併存させる形で「AIに聞く」機能を追加

---

## まとめ:目指す姿

> **来場者向けはシンプルなまま、運営者向けはAIネイティブ**

この二層構造が、Rohei Printer Systemのアイデンティティになりそう。

- 来場者(ライトユーザー):QR→いいね という直感的なUX
- 運営者(パワーユーザー):AIと対話して展示会全体を構築

「AIネイティブな展示会運営プラットフォーム」というポジショニングは、既存の競合(従来型SaaS的な管理画面)から明確に差別化できる強力なコンセプト。
