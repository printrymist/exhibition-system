# AI-Nativeアーキテクチャ設計メモ

Rohei Printer System への適用を念頭に整理した、AI-Native時代のアプリケーション設計についてのメモ。中島聡氏の「AI-Nativeビジネスアプリ・アーキテクチャ」を起点に、本システムへの示唆を抽出したもの。

---

## 1. コアとなる発想の転換

### 従来のSaaSアーキテクチャ
- UIが主役、その下にビジネスロジック、さらに下にDB
- APIは「あれば便利な連携用の入口」程度の二級アクセス
- 「アプリケーションの価値はUIにある」という90年代的価値観

### AI-Nativeアーキテクチャ
- **APIこそがプロダクトの本体**
- LLMとUIはどちらもAPIを呼び出す対等な「クライアント」
- ビジネスロジックとデータだけが唯一の正解、画面も会話も単なる「見せ方」の違い
- 「UIでできること」と「AIでできること」のズレが構造的にゼロになる

---

## 2. 不変条件 (Invariant) の考え方

### 定義
「システムが正しく動いている限り、いつ・誰が・どこから操作しても、常に成り立っていなければならない条件」。

### 普通のバリデーションとの違い
- バリデーション = 「入口で身分証を確認する警備員」
- 不変条件 = 「建物の中にいる人は必ず身分証を持っている、という保証された状態」

前者は門番がいる場所だけ守られるが、後者は建物のどこにいても成立している。

### なぜAI-Native時代に重要か
LLMが「APIを叩く一級クライアント」になると、UIを経由しない操作が日常的に発生する。不変条件がUI層にしか実装されていないと、LLMが操作した瞬間にルールが破られる。したがって **「不変条件はUIではなくAPI/DB層に置く」** という設計が必須になる。

### 実装すべき層 (Firebase スタックの場合)

| 層 | 役割 | 例 |
|---|---|---|
| Firestore Security Rules | 構造的なルール、権限 | 「作品ドキュメントには必ず `ownerId` が含まれる」 |
| Cloud Functions | ビジネスロジック的な不変条件 | 「無料プランの作品数上限」「作品削除時の関連リアクション削除」 |
| Firestore Transaction | 複数更新の原子性 | 「集計値と詳細レコードを必ずペアで更新」 |

**原則:** UIで止めるのは親切心、API/DB層で止めるのが不変条件。UIにもバリデーションは出すが、ルールの本体はもう一段奥にある二重構造にする。

## 3. Rohei Printer System への適用

### 守るべき不変条件の候補

#### (a) 作品とギャラリーの関係
- すべての作品は必ず一つのギャラリー (またはオーナー) に紐付いている
- 孤児になった作品が存在してはいけない
- 破ると権限管理や課金計算が壊れる

#### (b) QRコードの一意性
- 一つのQRコードは一つの作品にしか対応しない
- 複数作品が同じQRに紐付くと、スキャン時にどれを表示すべきか決まらない

#### (c) フリーミアム上限 (今後実装)
- 無料プランのユーザーは登録作品数が上限を超えてはいけない
- UIだけでチェックすると、APIを直接叩かれたらすり抜ける
- **必ず Cloud Functions または Firestore Security Rules で守る**
- 課金モデルの根幹に関わるため最優先

#### (d) 訪問者カウントの整合性 (フィンガープリンティング実装時)
- 一人の訪問者は同一作品に対して 1日1カウントまで
- ブレると分析ダッシュボードの数字が信用できなくなる

#### (e) リアクション・コメントとオーナーシップ
- リアクションは必ず有効な作品IDに紐付く
- コメントの編集・削除は投稿者本人 (またはギャラリーオーナー) のみ

### LLM機能の位置づけ

すでに作品画像のカテゴリ分類で Claude API を使用しているが、これは「LLMをAPIのクライアントとして使っている」状態。今後の AI 機能 (例: 来場者分析の自然言語質問応答) も同じ枠組みで実装する。

- ❌ 専用のチャットUIに固有ロジックを作り込む
- ✅ 既存の分析APIをツールとして Claude に渡す

こうしておけば「分析画面でできること」と「AIに聞いてできること」が常に一致する。

## 4. 設計原則 (Claude Code 作業時の指針)

1. **不変条件は Firestore Security Rules / Cloud Functions 層に置く。** UIのバリデーションは体験のためであり、ルールの本体ではない。
2. **新機能を実装するとき、まず「これは UI / API / LLM のどこから呼ばれても破られてはいけないか」を考える。** YESなら API/DB層に実装する。
3. **フリーミアム上限のような課金関連ロジックは UI だけで止めない。** Functions側で必ず再検証する。
4. **AI機能は専用UIではなく既存APIのクライアントとして実装する。** UI機能と AI 機能の能力差を作らない。
5. **「UIそのものが体験価値」の部分 (バーチャル展示など) はこの原則に縛られなくてよい。** UI中心のままで構わない。

---

## 5. 当面追わない論点

中島氏が論じている内容のうち、現フェーズで深追いしないもの:

- **GUI Chat Protocol / presentForm / presentChart などの汎用プレゼンテーション部品**
  - エコシステムと標準化の問題で、単一プロダクトの最適化が先
- **業界標準化された LLM 駆動UI**
  - 規模が出てからの議論
